# Terraform Infrastructure for Aether Platform

This directory contains Terraform configurations for provisioning the complete Aether platform infrastructure on AWS, including:

- **VPC** with public/private subnets across multiple availability zones
- **EKS Clusters** (management cluster for Aether services, workload clusters for tenants)
- **RDS PostgreSQL** instances for Aether API and Keycloak
- **ECR Repositories** for Docker images (API, UI, Agent)
- **IAM Roles** and security groups with least-privilege access

## Prerequisites

- [Terraform](https://www.terraform.io/downloads.html) >= 1.0
- [AWS CLI](https://aws.amazon.com/cli/) configured with appropriate credentials
- AWS account with permissions to create VPC, EKS, RDS, ECR, IAM resources
- `kubectl` for Kubernetes cluster access

## Quick Start

### 1. Configure Variables

```bash
# Copy the example variables file
cp terraform.tfvars.example terraform.tfvars

# Edit terraform.tfvars with your configuration
# At minimum, review and update:
# - aws_region
# - project_name
# - eks_public_access_cidrs (IMPORTANT: restrict in production)
```

### 2. Initialize Terraform

```bash
terraform init
```

### 3. Review Infrastructure Plan

```bash
terraform plan -out=plan.tfplan
```

Review the planned changes carefully before applying.

### 4. Apply Configuration

```bash
terraform apply plan.tfplan
```

This will provision the complete infrastructure (typically takes 15-20 minutes).

### 5. Configure kubectl

```bash
# Management cluster
aws eks update-kubeconfig --region us-east-1 --name aether-management-cluster

# Workload cluster
aws eks update-kubeconfig --region us-east-1 --name aether-tenant-1-cluster
```

## Security Best Practices

### State File Management

**CRITICAL:** Terraform state files contain sensitive information including:
- Database passwords (even when auto-generated)
- Resource IDs and endpoints
- IAM credentials and tokens

**Local Development:**
- State files (`*.tfstate`, `*.tfstate.*`) are automatically excluded from git
- Keep state files secure on your local machine
- Never commit state files to version control

**Production/Team Environments:**

Configure S3 backend for remote state storage:

```hcl
# backend.tf
terraform {
  backend "s3" {
    bucket         = "your-terraform-state-bucket"
    key            = "aether/terraform.tfstate"
    region         = "us-east-1"
    encrypt        = true
    dynamodb_table = "terraform-state-lock"
  }
}
```

Benefits:
- Centralized state storage
- State locking (prevents concurrent modifications)
- Encryption at rest
- Team collaboration
- Automated backups

### Variable Files

**terraform.tfvars:**
- Contains your actual configuration values
- Already in `.gitignore` - will not be tracked
- Keep this file secure and never share publicly

**terraform.tfvars.example:**
- Template with example values only
- Safe to commit to version control
- Use as reference for required variables

### Plan Files

**Plan files (*.tfplan) may contain sensitive data:**
- Generated by `terraform plan -out=plan.tfplan`
- Automatically excluded from git by `.gitignore`
- Delete after applying: `rm *.tfplan`

### Database Security

**Passwords:**
- Leave `aether_db_password` and `keycloak_db_password` empty in terraform.tfvars
- Terraform will auto-generate secure passwords
- Passwords stored in AWS Systems Manager Parameter Store (encrypted)
- Retrieved automatically by Terraform outputs

**RDS Configuration:**
- Databases deployed in private subnets (not publicly accessible)
- Security groups restrict access to EKS cluster nodes only
- Encryption at rest enabled by default
- Automated backups configured (7 day retention)
- Deletion protection enabled for production

### EKS Security

**Public Access Configuration:**

⚠️ **IMPORTANT:** The default example sets `eks_public_access_cidrs = ["0.0.0.0/0"]`

For production environments:
1. Restrict to your organization's IP ranges:
   ```hcl
   eks_public_access_cidrs = ["YOUR_OFFICE_IP/32", "YOUR_VPN_RANGE/24"]
   ```
2. Or disable public access entirely:
   ```hcl
   eks_endpoint_public_access = false
   ```

**Node Security:**
- Nodes deployed in private subnets
- NodePort range (30000-32767) exposed via security groups
- Consider using LoadBalancer services instead of NodePort for production
- Enable Istio/API Gateway for additional security layer

### AWS Credentials

**Never hardcode AWS credentials:**
- Use AWS CLI profiles: `aws configure --profile aether`
- Use environment variables: `AWS_PROFILE=aether terraform apply`
- Use IAM roles when running in CI/CD
- Terraform configurations use AWS provider authentication automatically

## Infrastructure Components

### VPC
- CIDR: 192.168.0.0/16 (configurable)
- Public subnets for NAT gateways and load balancers
- Private subnets for EKS nodes and RDS
- NAT gateways for private subnet internet access
- VPC endpoints for AWS services (optional)

### EKS Clusters

**Management Cluster:**
- Runs Aether API, UI, and Keycloak
- Default: 2 x t3.medium nodes
- Auto-scaling: 1-4 nodes

**Workload Cluster(s):**
- Run tenant workloads with Aether agents
- Default: 2 x t3.medium nodes
- Auto-scaling: 1-4 nodes
- Multiple tenant clusters supported

### RDS PostgreSQL
- Version: 15.4 (configurable)
- Separate instances for Aether and Keycloak
- Automated backups with 7-day retention
- Multi-AZ for high availability (production)
- Monitoring via CloudWatch

### ECR Repositories
- aether-api
- aether-ui
- aether-agent
- Scan on push enabled
- Lifecycle policies (keep 30 most recent images)

## Outputs

After applying, Terraform provides outputs for:
- EKS cluster endpoints and config
- RDS endpoints and connection strings
- ECR repository URLs
- Database credentials (SSM Parameter Store paths)

View outputs:
```bash
terraform output
```

Get specific output:
```bash
terraform output management_cluster_endpoint
```

## Accessing Secrets

Database passwords are stored in AWS Systems Manager Parameter Store:

```bash
# Aether database password
aws ssm get-parameter --name "/aether/production/db/aether/password" --with-decryption --query 'Parameter.Value' --output text

# Keycloak database password
aws ssm get-parameter --name "/aether/production/db/keycloak/password" --with-decryption --query 'Parameter.Value' --output text
```

These are automatically used in Kubernetes secrets when deploying services.

## Post-Deployment Steps

### 1. Retrieve Database Credentials

```bash
# View outputs
terraform output

# Get database endpoints
terraform output aether_db_endpoint
terraform output keycloak_db_endpoint
```

### 2. Build and Push Docker Images

See the root-level scripts:
- [scripts/build-and-push-api.sh](../scripts/build-and-push-api.sh)
- [scripts/build-and-push-ui.sh](../scripts/build-and-push-ui.sh)
- [scripts/build-and-push-agent.sh](../scripts/build-and-push-agent.sh)

```bash
# Set your AWS account ID
export AWS_ACCOUNT_ID=123456789012

# Build and push all images
../scripts/build-and-push-api.sh
../scripts/build-and-push-ui.sh
../scripts/build-and-push-agent.sh
```

### 3. Deploy to Management Cluster

```bash
kubectl config use-context aether-management
kubectl create namespace aether-system

# Deploy platform services using deployment scripts
# See scripts/build-and-push-*.sh and component-specific deploy scripts
```

### 4. Register Tenant Clusters

For each workload cluster, register in the Aether UI:
1. Navigate to Settings → Clusters
2. Add new cluster with kubeconfig
3. Install agent using the provided Helm chart

## Maintenance

### Updating Infrastructure

```bash
# Review changes
terraform plan

# Apply updates
terraform apply
```

### Destroying Infrastructure

⚠️ **WARNING:** This will delete all resources including databases (unless deletion protection is enabled).

```bash
terraform destroy
```

For production, ensure:
1. Databases have deletion protection enabled
2. State files are backed up
3. Data is backed up independently
4. Team is notified

## Troubleshooting

### State Lock Issues

If state is locked (concurrent apply/destroy):

```bash
# List locks (requires DynamoDB table)
aws dynamodb scan --table-name terraform-state-lock

# Force unlock (use with caution)
terraform force-unlock <LOCK_ID>
```

### Authentication Issues

```bash
# Verify AWS credentials
aws sts get-caller-identity

# Update kubeconfig
aws eks update-kubeconfig --region us-east-1 --name aether-management-cluster
```

### Plan Fails

```bash
# Refresh state
terraform refresh

# Re-initialize
rm -rf .terraform
terraform init
```

## CI/CD Integration

For automated deployments:

1. Store state in S3 backend
2. Use IAM roles for authentication
3. Use Terraform Cloud/Enterprise for additional features
4. Implement approval workflows for production
5. Use separate workspaces for environments

Example GitHub Actions:
```yaml
- name: Terraform Apply
  env:
    AWS_ACCESS_KEY_ID: ${{ secrets.AWS_ACCESS_KEY_ID }}
    AWS_SECRET_ACCESS_KEY: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
  run: |
    terraform init
    terraform plan -out=plan.tfplan
    terraform apply plan.tfplan
```

## Cost Estimates

Approximate monthly costs (us-east-1):

- VPC & Networking: $32-96 (NAT Gateways)
- RDS (2x db.t3.micro): $30
- EKS Control Planes (4 clusters): $292 ($73 each)
- EC2 Nodes (10x t3.medium): $380
- ECR: ~$1 (storage + transfer)
- **Total**: ~$735-799/month

**Cost Optimization Tips**:
- Use spot instances for non-production workloads
- Reduce node counts during off-hours
- Use single NAT Gateway (not recommended for production)
- Downsize RDS instances if workload allows
